# Auto Issue Solver Mode

## Overview

A third operational mode where gf-lt operates as an autonomous coding agent focused on solving a single issue from start to PR creation. The mode is self-contained and relies on bundled tools (bash, file editing, git via CLI) - no MCP dependency.

## Design Principles

1. **Session = One Issue** - Each mission is isolated with a fresh context
2. **No MCP Dependency** - Uses bundled tools only (bash, git, file ops)
3. **PM Supervisor** - Project manager agent provides periodic guidance
4. **File-Based Issues** - Issues stored as JSON in configurable directory structure
5. **Resumable** - Long operations can be checkpointed and resumed
6. **Graceful Exit** - `create_pr` tool or failure threshold (3 consecutive failures)

## Issues Directory Structure

```
{issues_dir}/           # Config: issues_dir, default ./issues (added to .gitignore)
├── open/               # Available for picking
│   ├── 5.json
│   └── 12.json
├── in_progress/        # Currently being worked on
│   └── 7.json
├── review/             # PR created, awaiting merge
├── done/               # Successfully merged
└── archive/            # Completed or rejected
```

## Issue JSON Schema

```json
{
  "version": "1.0",
  "id": "5",
  "title": "Fix login timeout bug",
  "description": "Full issue description in markdown...",
  "status": "open",
  "project_path": "/home/user/projects/myapp",
  "branch_name": "fix/issue-5-login-timeout",
  "labels": ["bug", "priority-high"],
  "priority": "high",
  "created_at": "2026-05-10T...",
  "updated_at": "2026-05-14T...",
  "related_issues": [],
  "acceptance_criteria": [
    "Login timeout reduced to 30s",
    "Error message shown on timeout"
  ],
  "context_files": ["src/auth/login.go", "tests/auth_test.go"],
  "comments": [
    {
      "author": "solver",
      "body": "Starting work on this",
      "created_at": "2026-05-14T..."
    }
  ]
}
```

**Notes:**
- `branch_name` is generated by LLM when starting work (per `docs/issue-workflow.md` instructions)
- `project_path` is the absolute path to the codebase
- `related_issues` replaces sub-issues (files stay simple)
- Comments accumulate during solver's work

## Mode Flag Structure

```bash
gf-lt --mission                              # Enter mission mode
gf-lt --mission --issue-id 5                 # Process specific issue (optional, auto-picks if not provided)
gf-lt --mission --agent-card ./card.json     # Custom agent personality (optional)
gf-lt --resume ./mission-state.json          # Resume from checkpoint
gf-lt --pm-interval 75                        # PM check-in every N tool calls (default: 75)
gf-lt --max-failures 3                        # Consecutive failures before abort (default: 3)
gf-lt --checkpoint-file ./checkpoint.json    # Custom checkpoint path
gf-lt --output json                           # Structured JSON output
gf-lt --quiet                                 # Suppress tool call logging
gf-lt --issues-dir ./issues                   # Directory containing issues (default: ./issues)
gf-lt --mission-tools                         # Enable mission tools in non-mission mode
```

**Config additions:**
```yaml
issues_dir: ./issues  # Relative to project or absolute (default: ./issues)
mission_tools_enabled: false  # Enable mission tools outside mission mode
```

## Mission Tools

Tools are automatically registered when `--mission` is used, or via `--mission-tools` flag.

| Tool | Args | Description |
|------|------|-------------|
| `move_issue` | `status` | Move issue to different status (review, done, archive) |
| `create_issue` | `id`, `title`, `description`, `branch_name` | Create a new sub-issue file |
| `create_pr` | `title`, `body`, `base` | Mark session complete, signal success |
| `pm_consult` | `question` | Request PM guidance (injects into conversation) |
| `add_issue_comment` | `body`, `author` | Add comment to issue file |

**Bundled tools (always available via `bash`):**
- `bash` - Execute shell commands (git, npm, make, etc.)
- `file_edit` - Modify existing files
- `grep`, `find`, `cat` - File operations

## Core Components

### 1. Mission Controller

Orchestrates the session lifecycle:
- Initialize context (load agent card, issue, workflow docs)
- Run main agent loop
- Trigger PM check-ins
- Handle termination (success, failure, manual interrupt)

### 2. PM Supervisor Agent

**Role**: Project Manager that provides context, guidance, and keeps the agent aligned with goals.

**Trigger Conditions**:
- Every N tool calls (configurable, default 75)
- When agent calls `pm_consult` tool
- On consecutive failures (2nd and 3rd before abort)

**Behavior**:
- Reads issue file (full content in context before replying)
- Receives: current issue, branch name, recent commits, tool call history (last 10)
- Responds with text guidance (free-form)
- Can say "you seem off-track, review the issue" or "good progress, focus on tests"

**Implementation**: Spawned as separate `chatRoundChan` request with PM role.

### 3. Issue Solver Agent (Main)

**System Prompt Source**:
1. Custom agent card (`--agent-card`)
2. Default bundled agent card: `sysprompts/auto-solver-default.json`

**Agent Card Structure**:
```json
{
  "id": "issue-solver",
  "role": "Senior Developer - Autonomous Issue Solver",
  "description": "An expert programmer who independently solves issues",
  "workflow_docs": "docs/issue-workflow.md",
  "workflow": {
    "steps": [
      "Read and understand the issue",
      "Plan implementation approach",
      "Create feature branch (generate branch_name)",
      "Implement in logical commits",
      "Write/run tests",
      "Create PR"
    ],
    "guardrails": {
      "max_commits": 20,
      "require_tests": true,
      "block_patterns": ["git push --force", "rm -rf /"]
    }
  },
  "pm_consult_tool": true,
  "create_issue_tool": true
}
```

## Session Lifecycle

```
┌─────────────────────────────────────────────────────────┐
│ INITIALIZE                                             │
│ ├─ Load agent card                                     │
│ ├─ Select issue:                                       │
│ │   ├─ Use --issue-id if provided                     │
│ │   └─ Else auto-pick first from open/                │
│ ├─ Read issue JSON                                    │
│ ├─ Move issue: open/ → in_progress/                  │
│ ├─ Generate branch_name (if not present)             │
│ ├─ Create/checkpoint session state                   │
│ └─ Inject initial context into agent                 │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ MAIN LOOP                                              │
│ ┌─ Agent sends message/tool calls                     │
│ │                                                    │
│ │  [tool call] ──→ Execute tool                      │
│ │       │                                           │
│ │       ↓                                           │
│ │  [increment tool call counter]                     │
│ │       │                                           │
│ │       ├─ [PM interval reached?] ──→ PM check-in   │
│ │       │   └─ PM reads issue file first             │
│ │       │                                           │
│ │       ├─ [tool == create_pr?] ──→ SUCCESS         │
│ │       │                                           │
│ │       └─ [tool == move_issue?] ──→ Update dirs    │
│ │                                                    │
│ └─ Agent continues                                     │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ TERMINATE                                              │
│ ├─ On SUCCESS: Move to review/, export chat, exit 0   │
│ ├─ On FAILURE: Move to archive/, export chat, exit 1  │
│ ├─ On INTERRUPT: Checkpoint, pause                     │
│ └─ On RESUME: Reload state, continue loop              │
└─────────────────────────────────────────────────────────┘
```

## Error Handling

| Condition | Response |
|-----------|----------|
| Branch exists | LLM sees error, attempts resolution |
| 3 consecutive failures | Force exit, checkpoint, export chat |
| LLM API error | Retry with backoff, checkpoint after 3rd |
| Network timeout | Retry, then checkpoint and pause |

**Consecutive failures** count when:
- Tool execution fails (bash returns non-zero)
- Git conflict resolution fails
- Test suite fails

## Checkpoint/Resume System

**Checkpoint File Format** (`mission-state.json`):
```json
{
  "version": 1,
  "issue_id": "5",
  "issue_path": "./issues/in_progress/5.json",
  "project_path": "/home/user/projects/myapp",
  "branch_name": "fix/issue-5-login-timeout",
  "agent_card_id": "issue-solver",
  "conversation": [...],
  "tool_call_count": 147,
  "consecutive_failures": 0,
  "commits_made": ["abc123", "def456"],
  "created_at": "2026-05-14T...",
  "updated_at": "2026-05-14T..."
}
```

**Resume Behavior**:
- Load checkpoint on `--resume` flag
- Re-read issue file (in case it was modified externally)
- Continue from saved conversation state

## Output Formats

**On Completion**:
- Exit code 0 (success) or 1 (failure)
- Console output: summary of commits, PR URL
- Chat export: `mission-{issue_id}-{timestamp}.json`

**Structured JSON Mode** (`--output json`):
```json
{
  "status": "success|failed|aborted",
  "issue_id": "5",
  "branch_name": "fix/issue-5-login-timeout",
  "commits": ["abc123", "def456"],
  "tool_calls": 203,
  "session_duration": "5m32s",
  "chat_export": "./mission-5-20260514.json"
}
```

## User Interaction During Mission

**Observability**:
- Real-time tool call logging (muted with `--quiet`)
- Progress: "Tool 147 | Commits: 5 | Failures: 0"

**Manual Override**:
- `SIGINT` (Ctrl+C): Checkpoint and pause
- `SIGTERM`: Checkpoint and abort
- Interactive prompt: "Resume, Abort, or Inspect?"

**Chat Input** (optional):
- User can type while mission running
- Injected as message from "user" role
- Can provide clarification or redirect

## Batch Mode

External script orchestrates multiple missions:
```bash
for id in 5 12 23; do
  gf-lt --mission --issue-id $id
done
```

Each iteration is a fresh session with isolated context.

## Remaining Work

1. ~~**Working directory not effective**~~ — **FIXED**: `execSingle()` now sets `cmd.Dir = cfg.FilePickerDir`, so all system commands (ls, grep, find, git, etc.) run in the project repo directory. `cd` updates FilePickerDir and all commands respect it.

2. ~~**`cd` path-doubling**~~ — **FIXED**: `FsCd` now resolves absolute paths directly and relative paths against FilePickerDir (like a real shell). Added idempotency guard: if already in target directory, returns early without error.

3. **PM agent empty response** — `PMAgentChat` can return empty string when the LLM returns no content, resulting in `[Empty Response - PM Guidance]` being injected as guidance. Need a fallback message when PM content is empty.

4. **`(task: in progress)` prefix on tool responses** — deprecated task system prints this prefix on every tool result, confusing the LLM. Should be removed.

5. **`findCall` false positives** — `ToolCallRE` regex matches `__tool_call__...__tool_call__` in any LLM text. Should be tightened or validated (check that captured content starts with `{`).

6. **End-to-end testing** — mission mode tests require a running llama.cpp server with a proper tool-capable model. Without it, the LLM either returns empty responses or generates malformed tool calls. A smoke test with mock/stub would allow CI-friendly validation, but a real end-to-end run needs the server.
